# 🧠 現状ロジックの整理

`ivs_publisher_controller.js` では既に：

```js
this._resumable =
  (this._boothStatus === "live" || this._boothStatus === "away") &&
  sessionStorage.getItem(this._resumeKey()) === "1"
```

つまり：

* booth.status が live/away
* かつ sessionStorage に resume フラグがある

場合だけ「復帰」表示になります。

---

# ❗ 問題

回線切断などでページが落ちた場合：

* `turbo:before-cache` が走らない
* `disconnect()` も呼ばれないケースがある
* よって resumeKey が保存されない

→ `_resumable` が false になり
→ 「配信開始」になる

---

# ✅ 改善方針（推奨）

**booth.status が live / away なら、常に復帰候補とみなす**

つまり：

* sessionStorage に依存しない
* Rails 側状態を正とする

---

# 🔧 修正内容（JSだけ変更）

`connect()` 内のこの部分を：

```js
this._resumable =
  (this._boothStatus === "live" || this._boothStatus === "away") &&
  sessionStorage.getItem(this._resumeKey()) === "1"
```

↓ に変更：

```js
// ★ boothが live/away なら常に復帰候補
this._resumable =
  (this._boothStatus === "live" || this._boothStatus === "away")
```

そして下の2行は削除：

```js
sessionStorage.removeItem(this._resumeKey())
this._resumable = false
```

（startBroadcast の最後にある部分）

# 🎯 こうすると

| 状態      | ボタン表示 |
| ------- | ----- |
| offline | 配信開始  |
| live    | 配信に戻る |
| away    | 配信に戻る |
| standby | 配信開始  |

完全に自然なUXになります。

# ⚖ なぜこれで安全？

* 実際に stage join するまでは broadcasting=false
* startBroadcast 内で通常 join する
* Rails 側は既に live 状態
* 不正二重開始は token API で 409 になる

設計的にも問題ありません。
