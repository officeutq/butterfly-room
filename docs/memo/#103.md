## #103 土台（AWS/IAM/EC2）やること一覧

### 1) まず決める：EC2 から AWS API を呼ぶ方式

方針はこれでOKです（本番で一番事故が少ない）：

* **EC2のIAMロール（Instance Profile）で認証**
* Rails / Docker コンテナは **環境変数にアクセスキーを持たない**

以降はこの前提で書きます。

## 2) IAM Policy を作成（IVS Real-Time 用）

**目的**：EC2 ロールに最小権限を付与し、`CreateStage` と `CreateParticipantToken` が通る状態にする。

### 最小ポリシー例（まずはこれ）

> ※ resource を絞るのは後でもできます。まずは疎通優先で “*” でOK（フェーズ1土台）。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "IvsCreateStage",
      "Effect": "Allow",
      "Action": [
        "ivs:CreateStage",
        "ivs:GetStage"
      ],
      "Resource": "*"
    },
    {
      "Sid": "IvsCreateParticipantToken",
      "Effect": "Allow",
      "Action": [
        "ivs:CreateParticipantToken"
      ],
      "Resource": "*"
    }
  ]
}
```
ポリシー名：`ButterflyRoomIvsRealTimeStagePolicy`

* この時点では **DeleteStage は付けない**（ガベコレしない方針なので）

  * ただし「疎通テストで作った stage を片付けたい」なら後述の手動削除だけでOK

---

## 3) EC2 の IAM ロールにアタッチ

* EC2 に紐付いている **IAMロール**（インスタンスプロファイル）を特定
* そのロールに、上で作った Policy をアタッチ

ここが終わると、**EC2上のRailsから AWS SDK 呼び出しが AccessDenied で落ちなくなる**のが狙いです。

---

## 4) 本番コンテナに `AWS_REGION` を渡す

あなたのコードは `ENV.fetch("AWS_REGION", "ap-northeast-1")` なので、基本は無くても東京になりますが、**明示しておくのが安全**です。

* `AWS_REGION=ap-northeast-1`

（ECSじゃなくEC2+dockerでも、`.env` / compose / systemd / どこかに入れておけばOK）

## 5) 疎通確認（ここが #103 土台のDone判定）

**目的**：Railsコンテナから `Aws::IVSRealTime::Client` が動くことを確認する。
この段階では UI や join は不要、**SDKが叩ける**ことだけ確認します。

### 5.1 STS で「ロールが効いてる」確認（最重要）

EC2上で、Railsコンテナから実行（あなたのルール通り docker compose 経由）：

```bash
docker compose -f docker-compose.production.yml exec app bin/rails runner 'p Aws::STS::Client.new.get_caller_identity.to_h'
```

期待：

* `Arn` が **EC2のIAMロール**（assumed-role/〜）になっている
* ここで落ちるなら、まずロール/ネットワーク/コンテナの認証経路が未達です

### 5.2 IVS Real-Time の CreateStage が通るか確認

土台なので一回だけ “テスト用 stage” を作って疎通確認します（ガベコレは今やらないが、手動削除は可）。

```bash
docker compose -f docker-compose.production.yml exec app bin/rails runner 'c=Aws::IVSRealTime::Client.new(region: ENV.fetch("AWS_REGION","ap-northeast-1")); r=c.create_stage(name: "probe-#{Time.now.to_i}"); puts r.stage.arn'
```

期待：

* ARN が出る（AccessDenied が消える）

> もしここで `Invalid Service In Action` 系が出るなら、IAMポリシーのアクション名が間違ってるか、別サービス名を指定してます（以前の “ivsrealtime が存在しない” みたいな事故はここで検出できます）。

### 5.3 （任意）GetStage で読めるか確認

上の ARN を使って：

```bash
docker compose -f docker-compose.production.yml exec app bin/rails runner 'arn=ARGV[0]; c=Aws::IVSRealTime::Client.new(region: ENV.fetch("AWS_REGION","ap-northeast-1")); p c.get_stage(arn: arn).stage.to_h' -- <STAGE_ARN>
```
