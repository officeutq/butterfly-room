## 結論：現状で OK？ → **ほぼOK。ただし 2点だけ直したい**

### 1) `Cast::BoothsController#status` の `booth` 変数が rescue で死ぬ可能性

いま `status` は `before_action :set_booth` してるのに、アクション内で `booth = Booth.find(params[:id])` を作り直してます。

そして `rescue => e` の中で `redirect_to cast_booth_path(booth)` を呼んでるので、**例外が Booth.find より前で起きた場合**に `booth` 未定義で二次エラーになり得ます。

**最小修正**：`booth` を作り直さず `@booth` を使う。

```ruby
def status
  StreamSessions::StatusService.new(
    booth: @booth,
    actor: current_user,
    to_status: params[:to]
  ).call

  respond_to do |format|
    format.html { redirect_to cast_booth_path(@booth), notice: "状態更新: #{params[:to]}" }
    format.turbo_stream { head :no_content }
    format.json { render json: { ok: true }, status: :ok }
    format.any { head :no_content }
  end
rescue => e
  respond_to do |format|
    format.html { redirect_to cast_booth_path(@booth), alert: e.message }
    format.turbo_stream { render plain: e.message, status: :unprocessable_entity }
    format.json { render json: { error: e.message }, status: :unprocessable_entity }
    format.any { render plain: e.message, status: :unprocessable_entity }
  end
end
```

### 2) `show.html.erb` に `canvas` target が2個ある（地味に危険）

Stimulus の targets は **複数マッチ**すると `this.canvasTarget` は「先頭」だけになります。
今は結果的に動く可能性が高いけど、後で UI いじった瞬間に事故るやつ。

```erb
<canvas class="w-100 rounded d-none" ... data-ivs-publisher-target="canvas"></canvas>
<canvas class="d-none" ... data-ivs-publisher-target="canvas"></canvas>
```

**どっちか消す**か、用途が違うなら target 名を分ける（例: `canvas` と `hiddenCanvas`）。

## Stage が増えないか？（質問の核心）

あなたの現状コードだと、

* `StartService` が **booth.ivs_stage_arn を stream_sessions.ivs_stage_arn にコピー**
* Token 発行は `@stream_session.ivs_stage_arn` を使う
* `EnsureIvsStageService` は **呼ばれてない**

なので、**Stage は増えません**（少なくともこの経路では）。
ログに `create_stage` が出ないのも正しいです。

> 逆に言うと、`EnsureIvsStageService` をどこかで呼び始めた瞬間に “セッションごと Stage 作成” に戻れるので、#124 で消しておく/封印しておくのはアリ。
