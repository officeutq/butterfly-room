## まず入れるべき差分（そのままコピペ可）

### 1) `Dockerfile.production`：assets:precompile を有効化

```dockerfile
FROM ruby:3.3-slim

ENV LANG=C.UTF-8 TZ=Asia/Tokyo \
    BUNDLE_PATH=/usr/local/bundle \
    BUNDLE_JOBS=4 BUNDLE_RETRY=3 \
    PATH="/usr/local/bundle/bin:${PATH}"

RUN apt-get update -qq && apt-get install -y --no-install-recommends \
    build-essential libpq-dev curl git nodejs npm \
    pkg-config libyaml-dev \
  && rm -rf /var/lib/apt/lists/*

WORKDIR /app

COPY Gemfile Gemfile.lock ./
RUN gem update --system -N && gem install bundler -N
RUN bundle config set deployment 'true' \
 && bundle config set without 'development test' \
 && bundle install

COPY package.json package-lock.json* ./
RUN test -f package.json && (npm ci || npm install) || true

COPY . .

ENV RAILS_ENV=production NODE_ENV=production

# ★ここが本命：ビルド中に必ず precompile
# credentials を要しないよう SECRET_KEY_BASE はダミーでOK（ビルド時のみ）
RUN SECRET_KEY_BASE=dummy bundle exec rails assets:precompile

CMD ["bash", "-lc", "bundle exec rails server -b 0.0.0.0 -p 3000"]
```

> ※ `assets:precompile` は **production で動く JS/CSS（propshaft 含む）を public/assets に吐く**ために必須です。

---

### 2) `docker-compose.production.yml`：静的配信をONにする

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile.production
    image: butterfly_room:prod
    container_name: butterfly_room-app
    working_dir: /app
    env_file:
      - ./.env.production
    ports:
      - "3000:3000"
    environment:
      RAILS_ENV: production
      NODE_ENV: production
      RAILS_SERVE_STATIC_FILES: "true"   # ★これを追加
    restart: unless-stopped
```

## 検証手順（原因切り分けにもなる）

### A. build 後に assets が「イメージ内に」あるか

```bash
docker compose -f docker-compose.production.yml build --no-cache app
docker compose -f docker-compose.production.yml run --rm app ls -la public/assets
```

ここで `public/assets` が存在してファイルが並べば **precompile は成功**。

### B. 実際に Rails が静的配信してるか（コンテナ内curl）

（コンテナ起動後）

```bash
docker compose -f docker-compose.production.yml up -d
docker compose -f docker-compose.production.yml exec app bash -lc "ls -la public/assets | head"
```

次に、外から（CloudFront ではなく ALB 直 or コンテナ直でもOK）：

```bash
curl -I http://<ALB or host>:3000/assets/application-*.css
```

* ここが **200 になる** → Rails側はOK。あとは CloudFront キャッシュや behavior の話
* ここが **404 のまま** → `RAILS_SERVE_STATIC_FILES` が効いてない / `public/assets` にそもそも無い

## 追加で確認したい “ハマりどころ”

### 1) `ENTRYPOINT` の存在

`Dockerfile`（開発用）には `entrypoint.sh` がありましたが、**Dockerfile.production には ENTRYPOINT が無い**です。
これは悪くない（db:prepare など余計なことしない）んですが、もし本番でも entrypoint 側で何かしていたつもりならズレます。

→ **本番は「assets はビルドで固定」**が方針なので、むしろ ENTRYPOINT で assets を触らない方が正解。

### 2) CloudFront が 404 をキャッシュしている

対策後も CloudFront 経由だけ 404 が残るなら、**`/assets/*` の invalidation** を入れるのが早いです。

---

## まとめ：#111 の「根本解」確定版

* production compose に volume が無いので **「volume 上書き対策」は現状すでに達成**
* 本命は
  ✅ **build 時 precompile**（Dockerfile.production）
  ✅ **静的配信ON**（RAILS_SERVE_STATIC_FILES）

この2点を入れて、上の検証 A/B で「イメージにある」「Railsが返す」を通したら、/assets 404 はほぼ止まります。

---

もし、**それでも `public/assets` が空**になる場合だけ次の追加調査に行きたいので、そのときは

* `bundle exec rails assets:precompile` のビルドログ（エラーの前後20行）
* `public/assets/.manifest.json` があるか（propshaftだと重要）

この2つだけ貼ってください。こちらで原因を一点突破します。
