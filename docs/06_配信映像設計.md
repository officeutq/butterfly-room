# 06_配信映像設計

## 目的

本設計書は、WebRTC（Amazon IVS Real-Time Stage）を用いた配信における  
**映像の解像度・アスペクト比・表示方法**を定義する。

特に以下を目的とする。

- publisher 側で配信映像の基準サイズを明確にする
- viewer 側では画面サイズに追随しつつ、映像の歪みや事故を防ぐ
- 今後の拡張（縦配信、解像度プリセット、描画方式変更）に耐える設計とする

---

## 基本方針（要約）

- 配信映像の **解像度・アスペクト比は publisher 側で確定** する
- viewer 側では配信映像を **画面サイズに追随させて表示** し、
  比率が合わない場合は **黒帯（レターボックス）** で吸収する
- 配信中に映像サイズ・比率を変更しない（スタンバイ〜配信開始時に確定）

---

## 用語

- **publisher**  
  配信者。getUserMedia / canvas / IVS Stage への publish を行う。

- **viewer**  
  視聴者。publisher が送出した映像トラックを受信・表示する。

- **viewer-frame**  
  viewer 側で映像を表示するコンテナ要素。  
  Bootstrap 等のレイアウトによりサイズ・配置が可変。

---

## Publisher 側：配信解像度・アスペクト比の決定

### 基本ルール

- 配信開始時に `getUserMedia()` で取得した **video track の実解像度**を基準とする
- ただし、配信負荷と帯域を考慮し **最大解像度の上限**を設ける
- アスペクト比は必ず維持する（歪ませない）

### 最大解像度の上限

| 向き | 最大解像度 |
|---|---|
| 横向き | 1920 × 1080 |
| 縦向き | 1080 × 1920 |

※ 正方形（例：1920×1920）固定は採用しない  
理由：
- 実映像の面積が減る（黒帯を大量に含む）
- エンコード・帯域コストが増える割に情報量が増えない

### 解像度決定フロー（概念）

1. `getUserMedia()` により video track を取得
2. 以下いずれかで実解像度を取得
   - `track.getSettings().width / height`
   - または `<video>` 要素の `videoWidth / videoHeight`（metadata 読み込み後）
3. 縦横比を維持したまま、上限（1920×1080 または 1080×1920）に収まるサイズへ縮小
4. 決定したサイズを **publisher canvas の width / height** に設定
5. `canvas.captureStream()` を用いて publish

### 注意事項

- 配信開始後に canvas のサイズを変更しない  
  （viewer 側の再レイアウト・瞬断を防ぐため）
- 高解像度になるほど CPU / 帯域コストが上がる  
  → 上限設定は必須
- 端末・ブラウザによっては `getSettings()` が未確定な場合があるため、
  `loadedmetadata` 後の `videoWidth/videoHeight` をフォールバックとして使用する

## Viewer 側：表示仕様（画面追随）

### 基本方針

- viewer 側は **受信した映像の解像度・比率を変更しない**
- 表示は viewer-frame のサイズに追随させる
- 比率が合わない場合は **黒帯（レターボックス）** で吸収する

### viewer-frame の考え方

- viewer-frame は Bootstrap 等のレイアウトによりサイズ・配置が可変
  - 横幅：コンテナ幅に追随
  - 高さ：vh / flex / 固定値 / ratio 等、画面設計に依存
- viewer-frame 自体が「表示枠」の責務を持つ

### 表示ルール（CSS）

```css
.viewer-frame {
  width: 100%;
  height: 100%;
  background: #000; /* 黒帯 */
  overflow: hidden;
}

.viewer-frame video {
  width: 100%;
  height: 100%;
  object-fit: contain; /* レターボックス表示 */
}
````

### 挙動

* 映像比率と viewer-frame の比率が一致する場合
  → 黒帯なしで最大表示
* 映像が横長・viewer-frame が縦長
  → 上下に黒帯
* 映像が縦長・viewer-frame が横長
  → 左右に黒帯

### 採用しない方式

* `object-fit: cover`
  → トリミングが発生し、意図しない映像欠けが起きるため不採用
* viewer 側で canvas に描き直して黒帯を付与
  → CPU負荷が高く、Phase1 では不要

---

## スタンバイとの関係

* 配信の前提条件（配信名・説明・映像条件）は **スタンバイ中に確定**する
* 配信解像度・アスペクト比も「配信開始時点で確定し、配信中は固定」
* スタンバイ中は viewer join 不可のため、
  解像度確定による事故（黒画面・途中変更）は発生しない
