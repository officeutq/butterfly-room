# README


**夜の店向け 配信 × 実店舗ハイブリッド売上プラットフォーム**

---

## 1. 企画背景

* コロナ以降、夜の店（キャバクラ／コンカフェ／ガールズバー）の客足が減少
* 娯楽は「外出」から「自宅完結型」へシフト
* 一方で、夜の店の価値である
  **距離感・指名感・双方向性**は、既存配信サービスでは代替できていない

---

## 2. コンセプト

> **夜の店を、アプリの中に再構築する**

* 来店を前提としない
* しかし「ただの配信」ではない
* 店舗文化・キャスト文化をそのまま持ち込む

---

## 3. サービス概要

### 基本構造

* 顧客は **ブース** に入室し、キャストの配信を視聴
* コメントによる双方向コミュニケーション
* アプリ内ドリンクを送信可能
* キャストがドリンクを「消化」した時点で売上確定

---

## 4. ブース設計（1対1／1対多の切替）

* ブースは店舗に所属
* ブースには複数キャストが参加可能
* ブース種別（MVP想定）

  * ホール（1対多）
  * ラウンジ（少人数）
  * プライベート（予約制）

👉 **距離感はブースで選ぶ**

---

## 5. 配信状態モデル

配信は以下の3状態のみを持つ。

| 状態  | 説明           |
| --- | ------------ |
| 未配信 | 配信終了／次回予定表示  |
| 配信中 | 通常配信         |
| 席外し | 一時離席（配信継続扱い） |

* 席外し中も配信は「終了」しない
* 未消化ドリンクは保持される
* 配信終了時に未消化ドリンクは一括返却

---

## 6. 課金・ポイント設計

### ポイント

* **1ポイント = 1円**
* 顧客は全店舗共通ポイントを所持
* ポイントは事前購入制

### ドリンク課金

* ブース内でワンクリック送信
* 送信時点では売上未確定
* キャストが消化した時点で売上確定

### 売上の流れ

```
顧客ポイント
 → 未消化ドリンク（保留）
 → 消化
 → 店舗売上ポイント
 → 手数料控除
 → 店舗へ支払い
```

---

## 7. KPI設計（思想）

### North Star KPI

> **店舗総売上（実店舗 + 配信）**

* 配信は来店誘導だけが目的ではない
* 配信単体で売上を作ることも正解

### 店舗管理者が見る指標（MVP）

* キャスト別

  * 配信売上
  * 配信時間
  * 配信売上 / 時間
  * 実店舗売上（参照）

👉 **評価・自動判定は行わない**

---

## 8. 役割と権限

### ユーザー種別

| ロール     | 役割         |
| ------- | ---------- |
| 顧客      | 視聴・課金      |
| キャスト    | 配信・消化      |
| 店舗管理者   | 設定・モデレーション |
| システム管理者 | 最終対応       |

---

## 9. モデレーション方針

* 運営は極力介入しない
* **店舗がモデレーター**

### 権限

* キャスト：ミュート／非表示
* 店舗管理者：店舗BAN／コメント削除
* システム管理者：全体BAN／不正対応

---

## 10. MVP画面構成

### 顧客

* トップページ

  * 配信中のお気に入り
  * 配信中のレア配信
  * 新着配信
* ブース（配信画面）
* 課金ページ

---

### キャスト

* ブースバックヤード（配信画面）

  * 配信開始／終了
  * 席外し切替
  * 未消化ドリンク消化
* 簡易ダッシュボード
* プロフィール編集

---

### 店舗管理者

* キャスト一覧
* 配信数値一覧
* メニュー設定（ドリンク／ブース／予約）

---

## 11. MVPでやらないこと

* 自動評価・ランキング
* 複雑なレコメンド
* 配信義務化
* AI分析
* 店舗介入型リアルタイム制御

---

## 12. 企画の核心（最終まとめ）

> Butterfly Room は、
> 配信と実店舗の両輪で、
> 店舗売上を最大化するための
> 夜の店向け配信プラットフォームである。
